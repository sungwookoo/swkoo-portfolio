# VISION

> 이 문서는 `swkoo-portfolio`의 **의사결정 기준선**입니다.
> 새 기능/리팩토링/카피 변경은 이 문서와 충돌하면 채택하지 않습니다.
> README 로드맵, `docs/REFACTORING_PROMPT.md`, 페이지 카피는 모두 이 문서로부터 파생됩니다.

---

## 1. 한 문장 목적

**"Commit → CI Build → Registry Push → Argo Sync → Pod Ready" 한 번의 배포 생명주기를 단일 운영자 시점의 한 타임라인으로 엮고, 그 위에 활성 알람을 겹쳐 운영 판단을 가속한다.**

이 앱은 **관측(observation)과 판단(decision)**만 담당한다. 조작(action)은 항상 원천 도구로 위임한다.

---

## 2. 고유 가치 — 이 앱이 없으면 잃는 것

Argo CD UI, Grafana, GitHub Actions는 각각 자기 구간만 안다. 배포 1건이 이 세 구간을 가로질러 진행될 때, 운영자는 탭 3개를 오가며 시간축을 머릿속에서 직접 이어붙여야 한다.

이 앱은 그 구간들을 **하나의 배포 이벤트로 엮어** 보여준다. 즉:

- 깃 커밋 1개 → 해당 커밋의 GitHub Actions run → 빌드된 이미지 태그 → Argo CD가 동기화한 Application 리비전 → 실제 Pod가 Ready가 된 시각 → 그 구간에서 발생한 알람.
- 이 연결이 끊어지는 순간(= CI 실패로 이미지 없음 / Sync는 됐는데 Pod가 CrashLoop / 배포 직후 5xx 상승)을 **한 화면에서** 발견 가능.

이 가치가 확립되지 않으면 이 앱은 Argo CD/Grafana의 중복이다.

---

## 3. Non-goals — 의도적으로 안 만드는 것

### ❌ Argo CD UI의 조작 기능 재구현 (최상위 금지)

- Sync 버튼, 수동 Refresh, 리소스 트리 편집, App 추가/삭제 같은 **조작 UI를 이 앱에 넣지 않는다.**
- 이유: 한 앱이 관측 + 조작을 모두 가지려 하면 Argo CD 재구현으로 미끄러지고, 그 길은 이미 Argo CD가 이긴다.
- 경계 신호: "Observatory에서 바로 Sync할 수 있으면 좋겠는데…" 같은 유혹이 오면 **Argo CD UI로 링크**만 연다. 재구현하지 않는다.

### 보조 Non-goals

- **Grafana 차트 재구현 금지**: CPU/Mem/지연 시계열 그래프는 이 앱에서 그리지 않는다. Grafana 패널 링크/임베드만.
- **오픈소스 제품화 금지** (현 단계): 남이 설치해서 쓰는 제품 지향은 범위 밖. Application/Repo 목록은 내 인프라 기준으로 하드코딩 유지. 설정 외부화는 필요성이 증명될 때 도입.
- **진짜 실시간 스트리밍 주장 금지**: WebSocket/SSE는 선택적. "실시간에 가까운(near-real-time)"을 정직하게 표기. Webhook + 짧은 폴링이면 충분.

## 4. 성공 기준 (완료의 정의)

**가중치: B(매일 쓰는 운영 탭) 70% / A(설계 설명) 30%**

### Primary — B: 내가 매일 먼저 여는 탭

- 배포/장애 대응 시 **Argo CD UI가 아니라 `/observatory`를 먼저 연다.**
- 체크 방법: 4주 연속, 주당 5회 이상 자발적 열람 기록(브라우저 방문 or 자체 기록).
- 둘이 충돌할 때(B vs A) **B가 우선.**

### Secondary — A: 3분 설계 설명

- 외부인(동료/면접관)이 `/observatory` 페이지를 3분 내에 읽고 **"왜 이렇게 설계했는지"**를 요약할 수 있다.
- 체크 방법: 실제 3명에게 보여주고 3분 타이머로 설명 요약받기. 3/3 성공 시 통과.

### 둘 다 실패 신호

- 이 앱이 있어도 Argo CD UI/Grafana 탭을 먼저 연다면 → B 실패. 기능이 아니라 **고유 가치 정의가 틀렸다.**
- 읽는 사람이 "기술 스택 전시"로 이해한다면 → A 실패. **내러티브를 고쳐야지 기능을 더 넣으면 안 된다.**

## 5. 경계선 — 이 앱 vs 주변 도구

| 도구 | 그 도구의 영역 | 이 앱의 관계 |
|---|---|---|
| **Argo CD UI** | CD 동기화, 리소스 트리, 수동 Sync/Refresh | 상태 **읽기**만. 조작은 링크로 위임 |
| **Grafana** | 런타임 메트릭 시계열 차트 | 패널 임베드/링크만. 차트 재구현 ❌ |
| **GitHub Actions** | CI 워크플로 로그, 재실행 | run 상태 **요약**만. 로그는 GitHub에서 |
| **Alertmanager** | 알람 규칙 관리, 라우팅 | **활성 알람 overlay**만. 규칙 편집 ❌ |
| **Observatory (이 앱)** | **배포 1건의 cross-tool 타임라인 + 활성 알람 overlay** | ← **유일하게 이 앱에만 있는 것** |

**기능 심사 질문 (모든 새 기능 제안에 적용):**
1. 이 기능은 Argo CD/Grafana/GitHub 중 하나가 이미 잘 하는가? → 그렇다면 만들지 않고 **링크**로 연결한다.
2. 이 기능은 위 세 도구 사이의 **경계를 잇는가?** → 그렇다면 이 앱의 핵심이다.
3. 이 기능은 조작(action)인가 관측(observation)인가? → 조작이면 거절한다.

---

## 6. 이 문서의 업데이트 규칙

- 성공 기준(§4)을 충족하지 못하고 4주가 지나면 **고유 가치(§2)를 재검토**한다. 기능을 더 넣어 해결하지 않는다.
- Non-goal(§3)을 깨뜨릴 때는 **왜 깨뜨리는지 이 문서에 기록**한 뒤에만 깬다.
- 한 문장 목적(§1)이 바뀌면 README 로드맵, REFACTORING_PROMPT, 페이지 카피를 **같은 PR에서 함께 갱신**한다.

---

## 7. 적용 범위

이 문서는 **Observatory(`/observatory`) 페이지의 정체성에만 적용**된다.

같은 인프라(swkoo.kr / k3s 클러스터) 위에서 운영될 수 있는 **별도 진입점**(예: 외부 사용자 대상 호스팅 데모 `/deploy`)은 자체 정체성·non-goal·성공 기준을 가진다. 그쪽 정체성이 정해지면 별도 문서(`docs/deploy-vision.md`)로 다루며, 본 문서의 §3 non-goals(특히 "오픈소스 제품화 금지")와 충돌하지 않는 범위에서 운영한다 — `/deploy`는 "내가 호스팅하는 친구용 데모"이지 "남이 설치해서 쓰는 제품"이 아니다.
